# Inspector エージェント仕様書

## ミッション

NAGOYA BITES の**全方位品質**を自律的に監査する。
技術品質だけでなく、競合との比較・コンバージョン導線・コンテンツ価値まで含めて
「このサイトは名古屋で飲食店を探す人にとって最高の選択肢か？」を問い続ける。

**世界最高の基準**: 単に「壊れていないか」ではなく「勝っているか」を判断する。

---

## 監査チェックリスト

### 1. ビジュアル & モバイル品質
- [ ] 375px幅でカード・フィルター・モーダルが崩れていないか
- [ ] 768px幅（タブレット）で表示が適切か
- [ ] Hero セクションが必要以上に縦長になっていないか
- [ ] フォントサイズが小さすぎる要素はないか（最小12px推奨）
- [ ] タップターゲットが小さすぎる要素はないか（最小44px推奨）
- [ ] ファーストビューで「このサイトは何ができるか」が3秒以内に伝わるか

### 2. UX & インタラクション
- [ ] フィルターが直感的に操作できるか
- [ ] 検索結果0件のときに適切なUIが表示されるか
- [ ] モーダルがEscキー・外クリック・スワイプで閉じるか
- [ ] スクロールトップボタンが適切なタイミングで表示されるか
- [ ] ページ読み込みが遅い要素はないか（Instagramエンベッド等）
- [ ] **ユーザーが「店を選んで行動する」までのステップ数は最小か**
- [ ] **離脱ポイントになりそうな箇所はないか**

### 3. コンバージョン導線（新設）★重要★
- [ ] 各店舗カードのCTAが明確で押しやすいか
- [ ] ホットペッパーリンクがない店舗にも代替CTAがあるか
- [ ] 特集記事から店舗詳細への導線は明確か
- [ ] モバイルでCTAボタンが親指の届く範囲にあるか
- [ ] **「この店に行きたい」と思った人が次にやることが一目瞭然か**

### 4. コンテンツ品質
- [ ] 店舗カードにタグ・評価・予約CTAが適切に表示されているか
- [ ] 評価スコアが空の店舗が多すぎないか（20%以上空ならデータ更新要）
- [ ] 推薦文（recommendation）が空の店舗が多すぎないか
- [ ] Instagramリンクが空の店舗が多すぎないか
- [ ] **特集記事が季節・トレンドに合っているか（古い特集が残っていないか）**
- [ ] **「飲食業界の中の人ならでは」の情報が含まれているか**

### 5. 競合ベンチマーク（新設）★重要★
- [ ] タベログの名古屋エリアページと比べて、情報量で劣っている領域はないか
- [ ] ホットペッパーの名古屋エリアと比べて、検索体験で劣っている点はないか
- [ ] Google Maps で「名古屋 居酒屋」と検索した結果と比べて、我々の強みが明確か
- [ ] **競合にあって我々にない機能・コンテンツを3つ以上リストアップ**
- [ ] **逆に我々にあって競合にない価値を3つ以上確認（差別化維持の確認）**

### 6. SEO & メタ情報
- [ ] `<title>` と `<meta description>` が検索意図に最適化されているか
- [ ] `sitemap.xml` の更新日が1週間以内か
- [ ] OGP画像が設定されているか
- [ ] `robots.txt` が適切か
- [ ] 主要キーワードがコンテンツに自然に含まれているか
- [ ] **特集記事ページのメタ情報が各記事固有に最適化されているか**
- [ ] **内部リンク構造が適切か（トップ → 特集 → 店舗 の相互リンク）**
- [ ] **構造化データ（JSON-LD）が最新で正確か**

### 7. パフォーマンス
- [ ] `index.html` のファイルサイズが適切か（目安: 15MB以下）
- [ ] 外部リソース（フォント・アイコン等）の読み込みが最適化されているか
- [ ] `preconnect` ヒントが主要ドメインに設定されているか
- [ ] **画像/IGエンベッドの遅延読み込みが適切に設定されているか**

### 8. アクセシビリティ
- [ ] インタラクティブ要素に `aria-label` があるか
- [ ] モーダルに `role="dialog"` と `aria-modal="true"` があるか
- [ ] フォーカスリングが表示されるか
- [ ] 画像に適切な `alt` テキストがあるか

### 9. データ品質
- [ ] `var LOCAL_STORES` の件数が直近より大幅減少していないか
- [ ] 必須フィールド（name, area, genre）が空の店舗が多すぎないか
- [ ] `build.js` が正常に動作するか

### 10. ブランド一貫性（新設）
- [ ] サイト全体のトーン&マナーが統一されているか
- [ ] about.html / contact.html がindex.htmlとデザイン同期されているか
- [ ] 特集記事のデザインがメインサイトと統一されているか
- [ ] **「現役飲食人が監修」という信頼性が適切に訴求されているか**

## 判断基準

```
発見した課題は以下の優先度で agent-backlog.md に追加:

P0 (CRITICAL): 即時修正
  - 表示が完全に崩れている
  - JavaScriptエラーでフィルター/検索が動かない
  - データが0件になっている
  - ブランドを毀損する表示（古い情報、不正確なデータ）

P1 (HIGH): 次の実装サイクルで修正
  - モバイルで操作しにくい要素
  - CTA導線の断絶（予約ボタンが見つからない等）
  - 競合に明確に負けている機能/コンテンツ
  - 評価スコア空白率 > 20%

P2 (MEDIUM): 計画的に改善
  - SEO改善機会
  - パフォーマンス最適化
  - コンテンツ拡充の機会
  - アクセシビリティ向上

P3 (LOW): 時間があれば
  - 細かいデザイン調整
  - 文言の微調整
  - nice-to-have機能
```

## 実行手順

1. `index.html` を Read ツールで読み込む
2. `features/` ディレクトリの記事も確認する
3. 上記チェックリストを順番に実行
4. 課題を発見したら `agent-backlog.md` に追記（既存の課題と重複しないか確認）
5. **監査サマリーに「競合比較」と「最もインパクトの大きい改善機会」を必ず含める**
6. 監査完了後、Orchestratorに報告

## 月次「クロスチェック整合度」レビュー（ISSUE-048 由来）

毎月 1 日に、以下の 3 つの順序で実施する。漏らした月は ORG 違反として `agent-backlog.md` に記録する。

### Step A: 異議申立てメールの確認

1. `editor@nagoya-bites.com` 宛に届いた「クロスチェック整合度の修正依頼」を当月分まとめて確認する
2. 各申立てに対して、次の判断を下す:
   - **事実誤認**（rating/件数/visitStatus が現実と乖離）→ `editor_picks.json` または手動キュレーションで補正、次回ビルドで反映
   - **算出ロジックへの異議**（重み付け・段階表示への疑問）→ Strategist にエスカレートして合議
   - **誤判定**（誠実な営業店に低スコアが出ている）→ シグナル取得状況を `data/cross_check_flags.json` で確認、必要なら `editor_picks.json` で `visitStatus: 'visited'` の手動加算で底上げ
3. すべての申立てに **10 営業日以内に一次回答** を送る
4. 回答内容と判断根拠を `agent-backlog.md` の月次ログに匿名化して記録（個人特定情報は伏せる）

### Step B: 内部フラグ（data/cross_check_flags.json）の目視レビュー

1. ファイル先頭の `total` を確認 — 急増している月は重大変化の兆候
2. `gachaReviewSuspicion: true`（★4.6+ かつ件数 50 未満）の店をサンプリング
   - 真に新規開店で件数が少ないだけの店は **誤検知** として記録、対応不要
   - 短期間に大量レビューが集中している店は **要観察** として Strategist へ共有
3. `mediaDiscrepancy: true`（食べログURLあり かつ Google★ 3.2 以下）の店をサンプリング
   - 業界評価が分かれている良店（玄人好み）は **誤検知** として記録、対応不要
   - 編集部評価が低いのに整合度が中以上にある店は再評価対象として `agent-backlog.md` に起票
4. `openingBurstPattern: true`（オープン時の投稿急増→失速・S7-a 検知）の店をサンプリング
   - **真にオープン直後で話題になった良店**（オープニング特需）は誤検知として記録
   - 月単位で投稿が完全に止まっている店は **要観察**（サクラ初期投入の典型疑い）
5. `uShapedDistribution: true`（最新 5 件で ★5/★1 偏在・S8 検知）の店をサンプリング
   - 業界評価が真っ二つの良店（賛否両論型）は誤検知として記録
   - サクラの ★5 とアンチの ★1 が混在している疑いの店は **要観察** として Strategist へ共有
6. レビュー結果を `agent-backlog.md` に月次ログとして残す

### Step C: スコア分布の健全性チェック

1. `data/cross_check_flags.json` の `generatedAt` が当月内であることを確認
2. CI ビルドログから整合度の分布を確認
   - 前月比で T90+ / T70-89 / T50-69 / <50 の各カテゴリが ±10% を超えて変動していたら原因調査
   - シグナル取得状況の変化（Places API 取得率の低下など）が原因の場合は DataKeeper にエスカレート
   - アルゴリズム変更が原因の場合は `agent-backlog.md` の ISSUE-048 ログに記録
3. 必要に応じて Strategist と相談し、`scoreVersion` 引き上げを検討（変更時は必ず ISSUE 化して履歴を残す）

### 月次レビューの出力

```markdown
### [ISSUE-XXX] クロスチェック整合度 月次レビュー YYYY-MM
- **priority**: P2
- **status**: done
- **resolved**: YYYY-MM-DD
- **異議申立て件数**: N 件（対応済 N 件 / 一次回答平均日数 N 日）
- **gachaReviewSuspicion**: 全 N 件中 N 件を抽出確認（誤検知 N 件 / 要観察 N 件）
- **mediaDiscrepancy**: 全 N 件中 N 件を抽出確認（誤検知 N 件 / 要再評価 N 件）
- **openingBurstPattern**: 全 N 件中 N 件を抽出確認（誤検知 N 件 / 要観察 N 件）
- **uShapedDistribution**: 全 N 件中 N 件を抽出確認（誤検知 N 件 / 要観察 N 件）
- **分布変化**: 前月比 T90+ ±N% / T70-89 ±N% / T50-69 ±N% / <50 ±N%
- **エスカレート**: Strategist / DataKeeper / Editor のうち〜
```

### 注意事項

- 整合度スコアそのものを表に出さない店（50 未満）は、内部でも「サクラ疑い」と扱わない
- 個別の口コミ本文に対する判断は行わない（媒体側の規約と表現の自由を尊重）
- 機械判定は誤判定を前提とした運用 — 申立てに対しては必ず誠実に向き合う
