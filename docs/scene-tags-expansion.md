# シーン分類の細粒度化 — 設計草稿

**status**: 草稿（ユーザーレビュー待ち / 設計判断要）
**作成日**: 2026-05-08
**ref**: ISSUE-035 / `docs/competitive-analysis-2026-05-06.md` A3 Catch-up

> このドキュメントは ISSUE-035「シーン分類の細粒度化」の設計提案。
> 競合（OZmall・ホットペッパー）の細粒度シーンに対して、NAGOYA BITES の Moat（業界視点）と矛盾しない範囲でどう拡張するかを整理する。
> 朝、ユーザーがレビュー → 採択・却下・修正のフィードバックを受けて Builder + Editor で実装。

---

## 1. 現状と Gap

### 現在のシーン

NAGOYA BITES のフィルター・特集記事で扱っているシーン:

```
デート / 女子会 / 接待 / 誕生日 / 宴会 / 大人数 / 個室 / 母の日 / GW
```

### 競合の細粒度シーン

| 競合 | シーン分類例 |
|---|---|
| **OZmall** | 記念日 → 誕生日 / 結婚記念日 / プロポーズ / 顔合わせ / 長寿祝い<br>女同士 → 女子会 / 推し活 / ママ会<br>男女 → デート / 合コン |
| **ホットペッパー** | カップルシート特集 / 大人の隠れ家 / カジュアル接待 / 大人数 / 同窓会 / 部署飲み |
| **一休** | 記念日 / 誕生日 / 接待・会食 / 夜景 / 個室 / アフタヌーンティー / プロポーズ |
| **OZmall 演出** | サプライズ / メッセージプレート / ハートケーキ / 花束 |

### 我々のスタンス

**追わない（Strategic Skip）:**
- 女性向け装飾演出（ハートケーキ・サプライズ写真撮影など）→ ターゲットを狭めすぎる
- 「合コン」型の煽り分類 → 編集独立性と矛盾

**追う（Catch-up）:**
- 利用シーンの実態に即した細粒度化（推し活・ママ会・両家顔合わせ・同窓会など）
- 業界視点で「店の選び方が変わるシーン」を新規追加

---

## 2. 拡張案：新規シーン 8 個

業界視点で「店の選び方が変わる」シーンを 8 個提案。各シーンに「業界人の読み方」を付与することで Moat との連動を担保。

### 推奨追加シーン

| # | シーン | 推奨タグ名 | 業界人視点の読み方 |
|---|---|---|---|
| 1 | 両家顔合わせ | `両家顔合わせ` | 完全個室＋静かな空間＋年配の方が落ち着くトーン。和食割烹が王道。料理の出る速度が読める店を選ぶ |
| 2 | 同窓会・歓送迎会 | `同窓会` | 20〜50名対応＋名前が呼びやすい店名＋会費明確。地元密着型の和食居酒屋が向く |
| 3 | 推し活・推しの誕生日 | `推し活` | デコレーション対応可＋SNS映え＋単独利用 OK。カウンター席のあるカフェ・ダイニング |
| 4 | ママ会・パパ会 | `ママ会` | 平日昼ランチ＋子連れ可（半個室）＋大人サイズの食器。和食ランチが鉄板 |
| 5 | 出張・接待二次会 | `接待二次会` | 22時以降営業＋全席個室＋静かな雰囲気＋アラカルト中心。バー・割烹が向く |
| 6 | 一人飲み・サク飲み | `ひとり飲み` | カウンター席＋常連客への配慮＋単価3,000円前後。大衆酒場・割烹が王道 |
| 7 | デート初回 | `デート初回` | カウンター並び＋初対面でも会話しやすい開放感＋ドレスコード明確 |
| 8 | プチ祝い・昇進祝い | `昇進祝い` | 半個室＋シャンパン・日本酒の選択肢＋写真記録不要のさりげなさ |

### 追加しないシーン（Strategic Skip）

| シーン | 追加しない理由 |
|---|---|
| プロポーズ | 装飾演出への過剰投資が編集独立性と矛盾 |
| 合コン | 編集トーンと不一致 |
| サプライズ写真撮影 | OZmall 型装飾、Moat 不一致 |

## 3. 実装方式の3案

### 案A: タグ層に追加（軽量・推奨）

`data/manual_stores.json` の各店舗にタグとして付与し、`index.html` のタグフィルターに新タグを追加。

**長所**:
- 既存 LOCAL_STORES の構造を壊さない
- フィルター UI の変更が最小限
- Editor が店舗単位で増減を制御できる

**短所**:
- 既存 4,584 店すべてには付かない（手動付与の店のみ）
- カバレッジが薄い

**工数**: 2〜3 日（タグ定義 + 主要 50 店付与 + UI 反映）

### 案B: 特集記事として独立（中規模）

各シーンを `features/scene-omoteashi-yorokai.html` のような特集記事として独立公開。タグ層は触らず、ロングテール SEO を狙う。

**長所**:
- ISSUE-031（ロングテール独自 KW）との相乗効果
- Editor 1 人で完結
- 記事ごとに業界人視点の長文解説が書ける

**短所**:
- 8 本作るのに延べ 80〜120 時間
- フィルターから引けない（記事から店へ）

**工数**: 8 本で 2〜4 週間

### 案C: シーン構造化 JSON（重量・将来案）

`data/scene_picks.json` を新設し、シーン × 店舗 × 業界人視点のマトリクスを構造化。`build.js` 経由で LOCAL_STORES にマージ。

**長所**:
- 業界視点シーン推薦が DB 化されて最強
- AI 引用候補化（構造化データとして強い）

**短所**:
- 設計工数が大きい
- editor_picks との重複・整合管理が必要

**工数**: 設計 + データ作成で 1〜2 ヶ月

## 5. リスクと注意点

| リスク | 対策 |
|---|---|
| タグ数が増えすぎてフィルター UI が破綻する | 新規タグは「シーン」カテゴリに集約。UI 折りたたみ要 |
| 業界視点なしのタグ羅列になり Moat が薄まる | 各シーンに「業界人の読み方」を必ず付与。タグ単体ではなく特集記事と連動 |
| 「推し活」「ママ会」が編集トーンに合わない | 編集規約と整合する形でトーン調整。装飾演出には踏み込まない |
| 既存 LOCAL_STORES のタグデータ構造が壊れる | manual_stores.json の追加のみ。LOCAL_STORES の構造は変えない |

## 6. ユーザー判断を仰ぎたい事項

1. **拡張するか / しないか**: シーン拡張は ISSUE-031（ロングテール独自 KW）との重複が大きい。本 ISSUE をクローズして ISSUE-031 に統合する選択もあり
2. **案A / B / C のどれを採用するか**: 推奨は案B 軽量版（特集 3 本のみ）→ 案A 部分実行 の段階的アプローチ
3. **追加シーンの優先順位**: 上記 8 シーンの中で特に優先したいものはあるか
4. **「ママ会」「推し活」を入れるか**: 編集トーンとの整合性を最終確認
5. **タグカテゴリ UI の刷新**: 新規タグ追加に合わせてフィルター UI を折りたたみ式にリニューアルするか
